# Matchmaking Enhancements: Age Constraints & Pagination

This document outlines the proposed implementation plan to upgrade the dating app's matchmaking engine. We will add `age`, `profession`, and `agePreference` to user profiles. Then, we will configure a fast compound index, scale the matching query logic to apply strict age bounds, and implement pagination for feed performance.

## User Review Required
> [!IMPORTANT]
> - By default, users without an `age` or `agePreference` in the database will need to be handled gracefully. This plan restricts matches to strictly valid configurations, meaning older users without an explicitly set age might be excluded from the new feed unless they update their profile.
> - The new `getAllUserProfiles` endpoint will now expect optional `?page=1&limit=10` query parameters in the URL for fetching older data to support infinite scrolling.
> - Please approve if this strict matching logic fits your current production data strategy!

## Proposed Changes

### Database Layer
#### [MODIFY] `src/models/user.model.js`
- Add `age` (Number, required) block.
- Add `profession` (String).
- Add `agePreference` (Nested Object with `minAge` and `maxAge` Numbers defaults).
- Append a compound index at the schema level:
  `userSchema.index({ gender: 1, age: 1 });`
  This ensures optimal database traversals when the user filters by their `interestedIn` (mapped to `gender`) and a range of bounds (`minAge` <= `age` <= `maxAge`).

### Controllers Layer
#### [MODIFY] `src/controllers/auth.controller.js`
- **`registerUser`**: Extract `age`, `profession`, `agePreference` from the request body. Enforce strict Mongoose validation checking during `User.create()`.
- **`getMyProfile`**: Augment the returned JSON payload to surface `age`, `profession`, and the mapped `agePreference` correctly.

#### [MODIFY] `src/controllers/user.controller.js`
- **`updateUserProfile`**: Append `"age"`, `"profession"`, and `"agePreference"` to the internal `allowedUpdates` constant so users can dynamically tweak their preferences dynamically via the settings UI.
- **`getAllUserProfiles`**: 
  1. Add pagination extracting `page` and `limit` variables safely defaulting to fetching `10` users per request.
  2. Implement `skip` and `limit` parameters on the main `User.find()` pipeline.
  3. Merge `agePreference` bounds into the MongoDB Query: `{ age: { $gte: currentUser.agePreference.minAge, $lte: currentUser.agePreference.maxAge } }`.
  4. Ensure robust edge-care logic if a user lacks the `agePreference` field.

## Open Questions

> [!WARNING]
> What should the default behavior be for `agePreference` if a user does not supply one during registration but the app updates require it? Our plan assumes `minAge: 18, maxAge: 100` as safety fallbacks unless you want to mandate users input it during sign up.

## Verification Plan

### Automated Tests
- Postman `POST /api/v1/auth/register` to verify new required fields create correctly matching users.
- Inspect MongoDB indexes via GUI to verify `{ gender: 1, age: 1 }` exists.

### Manual Verification
- Execute `GET /api/v1/users/profiles?page=1&limit=10` using a logged in account of age 21 with bounds 18-26. 
- Validate the feed successfully avoids 27+ year old users and successfully ignores user records where the gender does not match requirements.
